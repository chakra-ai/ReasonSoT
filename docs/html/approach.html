Approach & Solution Design

Why ReasonSoT exists, the key insight behind it, and the four ideas that do the real work.

The problem

A voice-based interview agent has three conflicting requirements:

  1. Feels natural — time-to-first-token (TTFT) must land under ~600 ms, or the conversation feels awkward.
  2. Thinks well — some questions genuinely need multi-step reasoning: system-design probes, trade-off discussions, adaptive follow-ups.
  3. Stays on rails — the interviewer has a purpose: cover topics at a set depth, adapt to the candidate, and not get lost in tangents.

A single-model setup forces a bad trade-off:

The key insight

Most interview turns don't need deep reasoning. Greetings, acknowledgments, topic transitions, and clarification requests are pattern-matchable. Only the turns that do need deep reasoning should pay the Sonnet-plus-thinking cost — and even then, the thinking budget should scale to the actual complexity of the question.

This is a classic dual-process framing — System 1 (fast, pattern-based, autonomous) vs. System 2 (slow, deliberate, resource-expensive). ReasonSoT applies it at the model-routing level.

The four pillars

Four ideas do the real work. Each pillar has its own design note; this section gives the intuition.

1. Zero-cost routing

Regex + signal scoring picks the path in microseconds. No LLM "planner" adding 200 ms.

2. Single-prompt reasoning modes

CoT / MoT / DST all stay in one API call. Breadth/depth happens in the thinking window, not in parallel calls.

3. Speculative CoT

Haiku drafts with a confidence score. Ship the draft if confident; Sonnet verifies otherwise.

4. Prefix-cached context

4 cache breakpoints on base / persona / KG / last-user. 90%+ cache read rate by turn 5.

Pillar 1 — Zero-cost routing

reason_sot/core/router.py decides between S1, S2, and speculative paths using only deterministic Python: regex pattern matches, a technical-term lexicon, conversation phase, and follow-up-chain depth. No LLM call. Latency: microseconds.

Every LLM call you can avoid is another ~200 ms of TTFT you don't spend. Routing via an LLM would reintroduce the latency we're trying to avoid.

    Score --> Gates{decision gates}
    Gates -- "simple pattern AND

    Gates -- "score ≥ thr − 0.2" --> D5["SPECULATIVE"]
    Gates -- "else" --> D6["S1, DIRECT"]

Pillar 2 — Single-prompt reasoning modes

Tree-of-Thought and similar multi-path reasoning frameworks traditionally fan out into N separate LLM calls and synthesize. That's latency- and cost-prohibitive for an interview.

ReasonSoT keeps all reasoning in a single LLM call and uses the thinking-token budget to simulate breadth/depth exploration:

Pillar 3 — Speculative CoT for borderline complexity

For utterances too complex for S1 but not clearly in S2 territory, we run a speculative path:

  1. Draft — Haiku generates a response and appends a self-assessed confidence score [CONFIDENCE: 0.X].
  2. Gate — if confidence ≥ 0.8, ship the draft. The user sees Haiku-speed TTFT.
  3. Verify — otherwise, Sonnet gets the draft plus a "verify and improve" prompt. Cheaper than generating from scratch.
sequenceDiagram
    participant U as User
    participant E as Engine
    participant H as Haiku (S1)
    participant S as Sonnet (S2)

    U->>E: utterance
    E->>H: draft with confidence prompt
    H-->>E: "Reply. [CONFIDENCE: 0.X]"

    alt confidence ≥ 0.8
        Note over E,H: 🟢 Fast path — ship draft
        E-->>U: draft (TTFT ~400ms)
    else confidence < 0.8
        Note over E,S: 🟡 Verify path
        E->>S: "verify and improve this draft"
        S-->>E: refined response
        E-->>U: refined (TTFT from draft, end from S)
    end

Figure 5 — Speculative CoT: most borderline turns ship at Haiku speed; only the genuinely hard ones pay for Sonnet.

Net effect: 48–66% latency reduction on medium-complexity turns compared to always using S2.

Pillar 4 — Prefix-cached persona + knowledge-graph context

Anthropic's prompt cache supports 4 ephemeral cache breakpoints per request. We place them strategically so that only the dynamic tail of each request needs fresh computation:

The knowledge graph itself is an in-memory, heuristic-built structure (no LLM call). Each turn updates nodes/edges from the user's utterance; the serialized summary goes into a cached system block so the model always has structured context about what's been discussed — without re-sending the full transcript uncached.

What this is — and isn't

ReasonSoT is:
  • A Python library for building voice-interview reasoning
  • Text-in, text-out with streaming — pair with ASR/TTS for full voice
  • Off-the-shelf Claude Haiku + Sonnet; all behavior comes from prompting, routing, and cache placement
ReasonSoT is not:
  • A voice stack (no ASR/TTS layer)
  • An agent framework (no tool use, no planning loop)
  • A fine-tuned model (no training pipeline)
